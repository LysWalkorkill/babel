1. Introduction

This memo defines how to use network management protocols in the My Teams project.

2. Conventions

The key words "MUST", "MUST NOT", "REQUIRED", "SHALL", "SHALL NOT", "SHOULD", "SHOULD NOT", "RECOMMENDED", "MAY", and "OPTIONAL" in this document are to be interpreted as described in BCP 14, RFC 2119 [RFC2119].

3. Overview

4. Client

4.1. Overview

The client must be able to connect to a server, to send multiple commands to it, to receive replies from it and to display the information to the user.

The client must send command in a specific format described later and should display the reply in a human readable format.

You can find commands and reply-codes in Commands Section(6) and Reply-Codes Section(7)

Upon connection, the client will receive a 200 OK response.

4.2. Send

The client must send commands to server in a particular format. Each command must be sent with a specific header as follows: 

          

struct Header __attribute__((packed))
{
    unsigned short command;
    int messageSize;
};

command: An unsigned short from this list corresponding to the command associated with the requests and responses. Allowed commands are: 

          

enum Response
{
    TRY = 100,
    RINGING = 180,
    OK = 200,
    USER_REGISTERED = 220,
    USER_LOGGED_IN = 221,
    USER_LOGGED_OUT = 222,
    CONTACTS = 240,
    INFO_CONTACT = 241,
    BAD_REQUEST = 400,
    UNAUTHORIZED = 401,
    NOT_FOUND = 404,
    INTERNAL_ERROR = 500,
    UNAVAILABLE = 503
};

enum Request
{
    INVITE = 1,
    ACK = 2,
    MESSAGE = 3,
    CANCEL = 10,
    BYE = 15,
    REGISTER = 20,
    LOGIN = 21,
    LOGOUT = 22,
    CONTACTS = 30,
    NOTIFY_CONTACT = 31,
    INFO_CONTACT = 32
};

messageSize: The total size in bytes of all arguments of the associated command after the header.

  • The header has a fixed size (6 bytes).
  • The client should verify the number and type of each argument of the associated command. There is another verification server side.
  • The header is followed by the command's arguments.

4.3. Receive

The client receives replies and requests from the server in a particular format. Each reply and request begins with a header (same as the client's header) and ends with the arguments.

The server uses the same requests and replies as the client.

5. Server

5.1. Overview

The server must be able to accept multiple clients, to send replies to a specific client, and to broadcast a message to multiple clients.

The server must be able to keep track of users. Any client with a valid user account must be able to access this data. The server should store the data so it can be reused after a restart.

5.2. Send

The server must answer clients' requests with replies. A reply starts with a header describing the reply (unsigned short: fisrt two bytes) and the size of the arguments (int: last four bytes). The header is followed by the arguments of the reply.

You may use the following structure for server replies: 

          

struct Header __attribute__((packed))
{
    unsigned short command;
    int messageSize;
};

command represents the reply code. Please refer to the Reply Code section for more information.

messaegeSize represents the size of the arguments. Arguments are detailed in the Reply Code section.

When receiving a new connection, the sever must send an 200 OK response.

5.3. Receive

When a client sends a request, the server receives a header with an unsigned short (2 bytes), representing the command, and an int (4 bytes) for the message size.

You may use the same structure used for replies.

The server should wait for the full message to be received before processing the request.

6. Requests

6.1. Invite

          
Send - to server

- {Header (Request: 1, messageSize: 6B)} + {userID: 4B, port: 2B}
            
        
          
Send - to client

- {Header (Request: 1, messageSize: 10B)} + {userID: 4B, ipv4: 4B, port: 2B}
            
        
          
Receive

- 100: Trying
- 400: BAD_REQUEST
- 401: UNAUTHORIZED
- 404: NOT_FOUND

6.2. Ack

          
Send

- {Header (Request: 2, messageSize: 0B)}
            
        
          
Receive

- 400: BAD_REQUEST
- 401: UNAUTHORIZED
- 404: NOT_FOUND

6.3. Register

          
Send

- {Header (Request: 20, messageSize: 100B)} + {username: 50B, password: 50B}
            
        
          
Receive

- 220: USER_REGISTERED + username
- 400: BAD_REQUEST

if request succeeded:
- 240: CONTACTS + {userId (4B) + status (1B) + username (50B)} * {users in db}

6.4. Login

          
Send

- {Header (Request: 21, messageSize: 100B)} + {username: 50B, password: 50B}
            
        
          
Receive

- 221: USER_LOGGED_IN + username (50B)
- 400: BAD_REQUEST

if request succeeded (per contact in DB):
- 240: CONTACTS + {userId (4B) + status (1B) + username (50B)} * {users in db}

6.5. Logout

          
Send

- {Header (Request: 22, messageSize: 0B)}
            
        
          
Receive

- 222: USER_LOGGED_OUT
- 400: BAD_REQUEST

6.6. Contacts

          
Send

- {Header (Request: 30, messageSize: 0B)}
            
        
          
Receive

- 401: UNAUTHORIZED

if request succeeded (per contact in DB):
- 240: CONTACTS + userId (4B) + status (1B) + username (50B)

7. Reply Codes

7.1. 1XX

The following reply codes are used for operations that are not done yet. The client or server will have to wait for another reply after receiving a 1XX reply.

100: Trying.

180: TRYING (sent by client).

  • port (2B): port that the other client should connect to.

180: TRYING (sent by server).

  • port (4B): ip that the client should connect to.
  • port (2B): port that the client should connect to.

7.2. 2XX

The following reply codes are used for success. They are sent to the user making the request. Here is a list of all of the success reply codes, and their meaning:

200: Command OK.

220: User registered.

  • username (50B): username of the registered user.

221: User registered.

  • username (50B): username of the registered user.

222: User logged out.

240: Contacts.

  • userId (4B): id of the user.
  • status (1B): 0 for disconnected, 1 for connected.
  • username (50B): username of the registered user.

7.3. 4XX

The followning reply codes are used for user errors. They are sent to the user from the server. Here is a list of all of the error reply codes:

400: Bad request.

401: Unauthorized.

404: Not found.

7.4. 5XX

The followning reply codes are used for server errors. They are sent to the user from the server. Here is a list of all of the error reply codes:

500: Bad request.

503: Unauthorized.
